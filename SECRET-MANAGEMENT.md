# 🔐 Secret Management Guide

Panduan untuk mengelola secrets dan environment configuration dengan aman, tanpa expose ke GitHub.

---

## 📋 Overview

Struktur environment management:

```
env/
├── .env.iam              ← Template (COMMIT to GitHub) ✅
├── .env.prod.iam         ← Production secrets (IGNORE, in .gitignore) ❌
├── .env.siimut           ← Template (COMMIT to GitHub) ✅
├── .env.prod.siimut      ← Production secrets (IGNORE, in .gitignore) ❌
├── .env.ikp              ← Template
├── .env.prod.ikp         ← Production secrets (IGNORE)
└── ...
```

### Rules:
- ✅ **Template files** (`.env.iam`, `.env.siimut`, etc) → **COMMIT to GitHub**
- ❌ **Production files** (`.env.prod.iam`, `.env.prod.siimut`, etc) → **NEVER COMMIT** (in .gitignore)

## 🚀 Quick Start

### 1. Clone Repository
```bash
git clone https://github.com/yourrepo/siimut-docker.git
cd siimut-docker
```

### 2. Generate Production Environment Files (Auto-generate Secrets)
```bash
# Generate IAM production env with auto-generated secrets
./prepare-iam.sh

# Generate SIIMUT production env with auto-generated secrets
./prepare-siimut.sh
```

The scripts will:
- ✅ Copy template from `env/.env.iam` → `env/.env.prod.iam`
- ✅ Auto-generate strong secrets:
  - `APP_KEY` (32-byte random, base64)
  - `IAM_JWT_SECRET` (32-byte random, hex)
  - Database passwords (16-byte random)
  - Passport RSA keys (2048-bit)
- ✅ Update `env/.env.prod.iam` with generated values
- ✅ **Never display secrets** in console (safe!)

### 3. Start Services
```bash
docker compose -f docker-compose-multi-apps.yml up -d
```

## 🔧 How It Works

### Template Files (Safe for GitHub)
`env/.env.iam` - contains:
```bash
APP_NAME="IAM Panel"
APP_ENV=production
APP_KEY=YOUR_APP_KEY_HERE                    # Placeholder (no real secret)
IAM_JWT_SECRET=YOUR_JWT_SECRET_HERE          # Placeholder
MYSQL_PASSWORD=change-me-in-production       # Generic default
...
```

### Production Files (Generated Locally)
After running `./prepare-iam.sh`:

`env/.env.prod.iam` - contains:
```bash
APP_NAME="IAM Panel"
APP_ENV=production
APP_KEY=base64:abc123def456...              # Real generated secret ✅
IAM_JWT_SECRET=abc123def456789...            # Real generated secret ✅
MYSQL_PASSWORD=xyzRandom16bytes==            # Real generated secret ✅
...
```

### docker-compose.yml Configuration
```yaml
services:
  app-iam:
    env_file:
      - ./env/.env.prod.iam  ← Uses production file with real secrets
```

## 🔐 Secrets Generated

### 1. APP_KEY (Laravel Application Key)
- **Length**: 32 bytes
- **Format**: base64 encoded
- **Generated by**: `php -r 'echo base64_encode(random_bytes(32));'`
- **Usage**: Encryption, hashing, sessions
- **Rotate**: `./prepare-iam.sh` with confirmation

### 2. IAM_JWT_SECRET (JWT Token Signing)
- **Length**: 32 bytes (64 hex chars)
- **Format**: hex string
- **Generated by**: `openssl rand -hex 32`
- **Usage**: Sign/verify JWT tokens between IAM and client apps
- **IMPORTANT**: Must be SAME for IAM and SIIMUT!

### 3. Database Passwords
- **Length**: 16 bytes
- **Format**: base64 encoded
- **Generated by**: `openssl rand -base64 16`
- **Usage**: MySQL root and user passwords

### 4. Passport RSA Keys (IAM only)
- **Type**: 2048-bit RSA keypair
- **Format**: PEM (standard)
- **Generated by**: `openssl genrsa -out private.key 2048`
- **Usage**: OAuth2 token signing (Passport)

## 🔄 Workflow Examples

### Scenario 1: New Developer Setup
```bash
# Developer clones repo
$ git clone ...

# Copy templates and generate secrets
$ ./prepare-iam.sh
$ ./prepare-siimut.sh

# Start developing
$ docker compose up -d

# Done! Each developer has unique secrets
```

### Scenario 2: Update Secrets/Rotate Keys
```bash
# Re-run to regenerate new secrets
$ ./prepare-iam.sh
# Answer 'y' when prompted to regenerate

# Restart containers to apply new secrets
$ docker compose restart app-iam
```

### Scenario 3: Production Deployment
```bash
# On production server, run prepare scripts
$ ./prepare-iam.sh
$ ./prepare-siimut.sh

# Docker compose uses env/.env.prod.* files automatically
$ ./deploy.sh
```

## ⚠️ Important Notes

### Critical!
- **Never commit** `.env.prod.*` files to GitHub
- **Never hardcode** secrets in code
- **Environment variables** override `.env.example` values

### Best Practices
1. ✅ Run `./prepare-*.sh` on FIRST setup
2. ✅ Keep `.env.example.*` files updated in Git
3. ✅ Different secrets per environment (dev/staging/prod)
4. ✅ Rotate secrets if ANY are exposed
5. ❌ Don't share `env/.env.prod.*` files via email/chat

### If Secret is Compromised
```bash
# 1. Revoke old secret immediately
# 2. Re-generate everything
$ ./prepare-iam.sh      # Choose 'y' to regenerate
$ ./prepare-siimut.sh

# 3. Restart containers
$ docker compose down && docker compose up -d

# 4. Update .env.example if default changed
$ git add env/.env.example.*
$ git commit -m "chore: update env template defaults"
```

## 🛡️ GitHub Safety Checklist

Before pushing to GitHub:
- [ ] All `.env.prod.*` files in `.gitignore`
- [ ] Template files (`.env.iam`, `.env.siimut`) have NO real secrets
- [ ] No API keys, passwords, or private keys in Git
- [ ] `.gitignore` reviewed and correct
- [ ] Ran `git status` to check nothing sensitive is staged

### Verify Before Commit
```bash
# Check what will be committed
$ git status

# Should show NO .env.prod.* files
# Should show template .env.* files only
$ git diff --cached env/

# Verify .gitignore works
$ git check-ignore -v env/.env.prod.*
# Should output: env/.env.prod.* is in .gitignore
```
